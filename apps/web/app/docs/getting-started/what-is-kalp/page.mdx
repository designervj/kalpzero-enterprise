# What is Kalp?

Kalp is the shared operating platform behind multi-tenant business websites, operator workspaces, and vertical business modules. It is not a separate codebase for each customer. Instead, one platform serves many agencies and tenants, with behavior shaped by business configuration, runtime documents, and role-scoped access.

<Callout title="Core idea" tone="brand">
  Kalp keeps the control plane, publishing shell, and vertical packs shared so a hotel, a commerce business, or a travel operator can ship quickly without forking the product.
</Callout>

<h2 id="platform-definition">Platform definition</h2>

KalpZero Enterprise is positioned as an enterprise-grade, multi-tenant operating platform for vertical businesses. In this repo, the public shell lives in `apps/web`, the domain API lives in `apps/api`, and background execution lives in `apps/worker`.

<h2 id="technical-topology">Technical topology</h2>

Kalp is easiest to understand technically when split into four lanes: ingress, orchestration, state, and rendered surfaces. That is the same pattern visible in your hand-drawn flow, just cleaned into a platform graph.

<ExecutionFlowDiagram />

The important engineering decision is that one runtime graph serves both operator actions and public website requests.

1. A platform operator enters through onboarding, provisioning, or admin flows.
2. Public traffic enters through the tenant-aware Next.js route shell.
3. FastAPI applies tenancy, permissions, and vertical rules before any runtime state is trusted.
4. Postgres, MongoDB, and Redis each handle a narrow class of responsibility instead of overlapping badly.
5. The resulting state is rendered back through one shared web app as a public website, dashboard, or studio preview.

That is why Kalp can look tenant-specific while still behaving like one product core operationally.

<h2 id="why-shared-platform">Why shared platform</h2>

The main design choice is to avoid cloning the product for every business. A shared platform means Kalp can enforce tenant boundaries, centralize fixes, and keep new modules compatible across many businesses.

- Agencies and tenants stay inside a common control plane instead of separate repos.
- Public pages can vary by tenant because routes, themes, and content are driven by runtime documents.
- Admin behavior can vary by tenant because modules, features, navigation, and visibility are shaped by the Business Blueprint.
- Future surfaces such as native mobile or other operator consoles can reuse the same contracts.

<blockquote>
  One tenant can publish a hotel booking site while another runs a commerce catalog, but both still sit on the same Kalp core.
</blockquote>

<h2 id="wave-one-verticals">Wave one verticals</h2>

The current wave one focus is India-first execution for hotel management, commerce, and tour and travel operations. Later verticals extend the same core instead of replacing it.

| Current focus | Why it matters |
| --- | --- |
| Commerce | Catalog, pricing, inventory, tax, payments, and fulfillment already fit the shared runtime model. |
| Hotels | Public hotel content, room operations, folios, staff scheduling, and stay records can live on one tenant contract. |
| Travel | Packages, transfers, activities, and hotel management flows can be combined in one platform operator shell. |

<h2 id="best-fit">Best fit</h2>

Kalp fits best when you want central governance plus tenant-level flexibility.

- Use it when many businesses should share one product core.
- Use it when the platform team wants fail-closed authorization and one operational control plane.
- Use it when public websites, admin workspaces, and provisioning should all come from the same contracts.
- Avoid it if every customer must own a totally separate product architecture with no shared runtime.
